01 / Overview
The 30-second summary

What we are building

A robot dog that autonomously walks through an office building and creates a photorealistic, interactive 3D map of the entire space — all computed onboard the robot in real time, with no external computer or wireless link needed.

Why it's novel

State-of-the-art 3D Gaussian Splatting SLAM systems have only ever been tested on desktop RTX 4090 GPUs. No published work has run this class of system on embedded hardware on a legged robot. We are closing that gap.

02 / Problem
What problem are we solving?

The gap in indoor mapping today

Creating accurate, photorealistic 3D models of indoor spaces has enormous practical value — from facility documentation and remote inspection to virtual tours and institutional record-keeping. But the tools available today are all flawed:

  • Matterport scanners — expensive proprietary hardware, costs thousands of dollars per unit
  • Manual sensor rigs — requires a trained technician walking the entire space
  • Traditional SLAM maps — not photorealistic; only produce point clouds or voxel grids, not navigable 3D scenes
Three specific unsolved problems
1

Embedded hardware — never been done

Gaussian-LIC and similar systems have only been tested on high-end desktop GPUs (RTX 4090). The Jetson Orin NX has a fundamentally different memory architecture — the CPU and GPU share the same 16 GB of RAM. The CUDA kernels need to be completely reworked. No published system has demonstrated this.

2

Challenging real-world environments

Most research uses controlled lab environments. Real office buildings have glass walls, shiny floors, and long corridors that look identical from different angles — all of which confuse SLAM systems. The WSO2 Colombo office is a perfect example of exactly this kind of challenging environment.

3

Coverage gaps from wheeled robots

Wheeled robots cannot complete full building coverage — they get stuck at door steps, floor level changes, and small obstacles. This leaves holes in the 3D map. A quadruped like the Go2 Edu can handle these situations naturally, achieving significantly better coverage.

The central research question: Can we get a state-of-the-art LiDAR-Inertial-Camera Gaussian Splatting SLAM system to run in real time on the onboard computer of a legged robot — and if so, how good is the resulting reconstruction?
03 / Solution
How the solution works

The core system: Gaussian-LIC

Gaussian-LIC (published at ICRA 2025 by Lang et al. from Zhejiang University) combines three technologies into one real-time SLAM pipeline:

LIDAR + IMU
FAST-LIO2

Handles pose estimation. Runs at 100 Hz even on ARM processors and stays stable through the robot's walking vibrations.

CAMERA
3D Gaussian Splatting

Handles the photorealistic appearance. LiDAR gives structure; camera gives colour. Together they build the navigable 3D scene.

COMBINED
Gaussian-LIC

The full fusion: LiDAR initialises the Gaussian map, IMU keeps it stable, camera gives it colour. All in real time.

Three-phase execution plan

Phase 1 — Simulation & Baseline (Weeks 1–8)

Build a simulation environment in Gazebo under ROS 2 that models all sensors on the Go2 Edu: Unitree L1 LiDAR, built-in IMU, Intel RealSense D435i, and Hesai XT16 LiDAR. Integrate Gaussian-LIC into simulation and get it working before touching the real robot. This produces the simulation video required by WSO2 to unlock access to the physical robot.

Phase 2 — Porting to Jetson Orin NX (Weeks 9–14)

The main technical challenge. Four key changes to run Gaussian-LIC on the Orin NX:

  • CUDA kernel rewrite — optimise for the Orin's Ampere GPU cores and unified memory architecture
  • Local Gaussian windowing — only keep Gaussians near the robot in active memory at any time
  • Resolution reduction — process camera at lower resolution to reduce computational load
  • Spawn-rate limiting — adjust how new Gaussians are added to avoid memory bottlenecks

Phase 3 — Real-World Testing at WSO2 (Weeks 15–18+)

Bring the robot to the WSO2 Colombo office (#105, Bauddhaloka Mawatha, Colombo 04). A team member drives with a controller while the onboard system builds the Gaussian map in real time. Routes include glass-walled corridors, open areas, and room entrances. Results are packaged as an interactive web viewer viewable in any browser.

06 / Workstreams
How we work in parallel

The core principle: decouple with interfaces

Every member works on something that produces a well-defined output. Others can build against a mock / simulated version of that output while the real version is being built. No one is blocked waiting for anyone else.

The five roles are not a pipeline — they are five independent learners for the first 6–8 weeks. Each person is learning their own domain and building something that stands alone. Integration only happens at defined merge points.
WEEKS 1 – 8

Parallel phase

All 5 members work completely independently. No one waits for anyone. Each person has a concrete solo deliverable due at week 8.

WEEKS 9 ONWARDS

Integration phase

The five solo deliverables are combined. Dependencies now exist — but by this point everyone has enough domain knowledge to work through them together.

Per-member solo deliverables (Weeks 1–8)
Integration merge points

Merge 1 — Week 8: WSO2 simulation gate

Member A's FAST-LIO2 + Member C's Gazebo simulation are combined. FAST-LIO2 now runs on simulated sensor data from Gazebo instead of the downloaded dataset. This is the video sent to WSO2 to unlock physical robot access. Members B, D, E continue their solo work unblocked.

Merge 2 — Weeks 10–11: Jetson + Physical rig

Member B's Jetson profiling work is combined with Member A's SLAM codebase to start porting Gaussian-LIC to the Jetson together. Member E delivers the physical sensor rig — Member C now has hardware to run real extrinsic calibration using Kalibr.

Merge 3 — Weeks 14–15: Full pipeline + field prep

The complete pipeline runs on real hardware. Member D plugs in their evaluation scripts and web viewer. This is the milestone for the WSO2 field visit booking.

07 / Performance Targets
What we are aiming to achieve
<0.3m
SLAM drift / 50m
FAST-LIO2 achieves 0.06 m over 81 m in comparable environments — our target is very conservative
>23 dB
PSNR — novel views
GS-LIVO hit 23.52 dB on the same Jetson Orin NX 16 GB hardware (IEEE RA-L 2025)
>5 Hz
Map update rate
GS-LIVO reaches ~20 Hz on same hardware — 5 Hz is a conservative baseline
>80%
Area coverage
Wheeled robot estimated at ~60% on same route — the legged platform closes that gap

Why these targets are achievable — not guesses

Each metric is anchored to published results from comparable systems, not estimated from scratch:

  • SLAM drift target — based on FAST-LIO2 published benchmarks (0.06 m over 81 m indoors). Our 0.3 m target is 5× more conservative.
  • PSNR target — based on GS-LIVO (IEEE RA-L 2025), which already ran on the exact same Jetson Orin NX 16 GB hardware and achieved 23.52 dB.
  • Map update rate — GS-LIVO hits ~20 Hz on the same hardware. Our 5 Hz threshold is a floor, not a ceiling.
  • Coverage — estimated from wheeled robot navigation studies on comparable office floor plans.
